# UI Before/After

Before:

- Settings navigation and page copy used "AI Verification Checks" framing.
- The policy registry was lifecycle-row oriented and could imply all listed lifecycle evidence families were actively verified.
- Policy IDs were too prominent for officer default scanning.
- Agent 1 pipeline copy did not clearly say checks are canonical by document type and did not clearly exclude sanctions/PEP/adverse-media screening.
- Application Review document rows exposed too much technical metadata in the default card.

After:

- Back-office nav/title uses "Document Verification Policies".
- Settings page explains that Agent 1 verifies document types consistently and workflows decide requirements, blockers, and follow-up triggers.
- Policy cards are document-first and show Active / Manual review only / Future / enterprise status.
- Policy IDs are retained in details/audit evidence, not the primary row.
- The Agent 1 pipeline describes "Identity & Document Integrity Agent" and explicitly states that Agent 1 does not approve, reject, waive, override, or perform sanctions/PEP/adverse-media screening.
- Application Review document rows are action-first: document name/type, verification status, issue/reason, blocks, next action, and officer buttons for details/accept/reject/request replacement.
- Routine technical/audit details remain available in expanded details.
- "Approval blocked" is no longer rendered with green pass styling in the touched document card.
