# P12-1 Phase A — regulated-record deletion discovery and protection plan

> **DISCOVERY ONLY — NO ENFORCEMENT CODE**
>
> This is a supervised classification and design artifact for DCI-001/DCI-003. It does not change deletion behavior, application logic, database constraints, foreign keys, migrations, data, staging, or production. It does not close P12-1 and makes no production-readiness claim.

**Repository baseline:** `main` at `d23cc45` (2026-07-10 checkout)

**Discovery date:** 2026-07-10

**Scope:** repository-static inspection only; no staging or production data was read or mutated

**Required approval before Phase B:** founder plus compliance/legal review of the classification, sanctioned contexts, and pilot boundary

## Executive summary

The concrete DCI finding is present and reachable. A portal client can delete a draft application through `ApplicationHandler.delete`; its shared cleanup helper explicitly deletes `sar_reports`, `compliance_memos`, `edd_cases`, `agent_executions`, `supervisor_audit_log`, `decision_records`, and normalized screening evidence before deleting the application. Separately, inline migration v2.13 runs during every boot and deletes EDD cases, compliance memos, decision records, the application, local files, and cascade children selected only by a company-name match. These are the exact DCI-001/DCI-003 anchors.

The next-highest risks are the non-production admin reset endpoint (`TRUNCATE ... CASCADE`/all-table `DELETE`), document deletion after evidence may already exist, unguarded manual/demo/smoke cleanup scripts, direct regulated-row deletion in shared test teardown, application/document/alert/review cascades, and the repository-defined S3 lifecycle rule that expires KYC objects after 2,555 days without coordinating DB disposition, legal holds, or purge evidence. There is no runtime client-account delete endpoint in the inspected baseline, and `fixture_filter.py` is read-only; however, the fixture package has no guarded destructive cleanup helper, so cleanup has leaked into generic scripts and test code.

The correct Phase B boundary is not “regulated table means delete always denied.” Ordinary and cascade deletion must fail closed, while a central, non-opt-in interceptor admits only validated capabilities for retention purge, future dual-control GDPR erasure, tightly scoped non-production fixture cleanup, and an exceptional migration/admin context. Retention purge from P12-8 must continue to delete retention-expired rows atomically with evidence. H2B remains wired-but-OFF and primarily anonymizes PII; P12-1 must not make future lawful erasure impossible.

## 2. Hard-delete inventory

Classification used below:

- **Regulated:** compliance, audit, decision, provenance, governance, or evidence data.
- **Sanctioned:** a controlled retention/erasure/fixture workflow with verified eligibility, appropriate environment/legal controls, and evidence.
- **Unsafe:** generic cleanup, ordinary parent deletion, startup repair, cascade, or a user/admin/test path that can remove regulated evidence without such a context.

## 7. Relationship to P10-7

P10-7 / RDI-013 is the database-permission belt-and-braces control: separate the migration/admin DB role from the runtime role, revoke runtime `UPDATE`/`DELETE` on `audit_log`, `decision_records`, and `supervisor_audit_log`, and stop cleanup code from deleting them.

P12-1 Phase B and P10-7 should divide responsibility as follows:

| Control | P12-1 Phase B | P10-7 |
|---|---|---|
| Classification | Defines regulated/hybrid/cascade-root sets and sanctioned capabilities | Consumes the approved append-only subset for grants |
| Runtime prevention | Central application DB-access interception before any mutation, including cascade-root awareness and structured denial telemetry | DB denies runtime-role UPDATE/DELETE even if application interception is bypassed |
| Sanctioned deletion | Validates retention/GDPR/fixture scope and evidence; names the required capability | Provides a separate privileged/admin role where deletion is legally allowed; runtime credentials never inherit it |
| Migrations | Does not grant generic boot cleanup; only reviewed migration-admin context | Owns role separation and GRANT/REVOKE deployment/rollback |
| Cascades | Pilot guard blocks parent delete before the cascade | Later constraint migration replaces unsafe cascades/restricts direct SQL |

Avoid duplicate/conflicting controls:

- P12-1 should not embed environment-specific DB usernames or GRANT logic.
- P10-7 should not independently invent a second regulated-table classification or sanctioned-context API.
- A manual retention purge of `audit_log`, if humans approve it, cannot run under the runtime role after P10-7; it must use the privileged retention/admin workflow.
- `supervisor_audit_log` remains never-purge. `decision_records` deletion requires an explicit legal-retention decision; the default is retain.

## 9. Proposed acceptance criteria for Phase B

1. Ordinary runtime credentials and application paths cannot hard-delete regulated evidence.
2. Unsafe application/client/document/party paths cannot erase audit, decision, screening, IDV, memo, EDD, RMI, monitoring, periodic-review, governance, or other evidence rows through explicit SQL or cascade.
3. Startup/boot schema and repair paths cannot delete regulated application evidence; the v2.13 company cleanup is absent from generic boot.
4. Fixture cleanup is an explicit command/helper, positively marker-scoped, limited to verified testing/staging or a proven throwaway DB, evidenced, and refused in production.
5. P12-8 retention purge still deletes eligible rows atomically with `data_purge_log` evidence and cannot use the capability for an ineligible table/row/cutoff.
6. Future GDPR erasure remains possible but the live path stays disabled until P9-1/PC-4; enabling requires verified DSAR correlation, legal arbitration, dual control, artifact handling, and evidence.
7. A denied destructive statement fails closed with a controlled typed error before any DB/file/S3 mutation.
8. Every denied attempt emits a structured security/compliance event with table, operation, caller/context, request/correlation ID, and outcome, without leaking SQL parameters or PII; an appropriate append-only audit sink records the event.
9. Multi-step mutations are atomic. Evidence-write failure rolls back sanctioned DB deletion; file/object disposition cannot leave the database falsely claiming a successful archive/purge.
10. Tests cover SQLite and PostgreSQL where possible, including direct SQL, dynamic SQL, comments/CTEs, quoted identifiers, `executescript`, `TRUNCATE`, cascade-root deletion, raw-cursor bypass, and concurrent context isolation.
11. Staging synthetic validation passes for denied ordinary deletion, allowed retention purge with evidence, refused production fixture cleanup, and allowed marker-scoped staging cleanup. No real staging record is used or deleted.
12. The implementation PR remains a controlled-pilot hardening claim only and makes no production-readiness or P12-1 closure claim until supervised validation and founder approval are recorded.

## 11. Phase B design requirements

The implementation review should reject these shortcuts:

- A flat global `REGULATED_TABLES` check that permanently denies all deletion, because it would break lawful retention and future erasure.
- Opt-in helper calls at selected endpoints while `DBConnection.execute` and parent cascades remain unguarded.
- `ENVIRONMENT != production` or `ENVIRONMENT == testing` as the sole fixture authorization.
- A context name supplied by request data, query parameters, CLI strings, or feature flags without an internal capability plus validation metadata.
- Logging a denial only to a mutable application log, or logging after partial mutation.
- Treating `documents`, `monitoring_alerts`, or `periodic_reviews` as entirely unregulated because they are business containers; their current rows contain regulated evidence.

Recommended sanctioned capability record (conceptual, not implemented here): context kind, actor/service identity, reason/legal basis, request/change/DSAR ID, approved scope, environment and verified DB identity, expiry, batch ID, second approver where required, and expected evidence sink. The interceptor should validate both the capability and the actual SQL target; the sanctioned engine must validate row eligibility.
