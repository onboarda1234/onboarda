# Onboarda staging fixture seeder

Path A: real-schema-adapted. **No schema changes.** All writes go to
columns that already exist in both `arie-backend/db.py`'s SQLite (dev)
and PostgreSQL (production/staging) schemas.

## Status

- staging-only
- idempotent
- audit-tagged (every fixture write is paired with a `fixture.*` row in `audit_log`)
- dry-run / apply modes
- generated REGISTER (Markdown table of seeded rows)
- bypasses the API layer (no screening replay, no notifications, no rule recompute beyond `compute_risk_score`)

## Pilot Canonical Dataset v1

The permanent 41-application pilot catalogue is separate from the legacy
`SCEN-*` fixtures. Its reviewed source is
`fixtures/pilot_canonical_dataset_v1.json`; its manifest, validator, guarded
seeder, and operator CLI are in the matching `pilot_canonical*` modules.
Nothing is seeded automatically.

From `arie-backend/`:

```bash
# No DB access or writes
python -m fixtures.pilot_canonical_cli validate
python -m fixtures.pilot_canonical_cli list

# Uses the configured DB, exercises one transaction, then rolls back. It also
# requires the intended Tier 0 runtime contract to already evaluate true; the
# CLI never changes that flag.
python -m fixtures.pilot_canonical_cli dry-run
```

The future apply operation is staging-only and requires
`ALLOW_PILOT_CANONICAL_SEED=1`, the literal confirmation token
`APPLY-PILOT-CANONICAL-DATASET-V1`, and the exact founder-reviewed manifest
SHA-256 via `--reviewed-hash`. See
`docs/pilot/PILOT_CANONICAL_DATASET.md` for the unexecuted staging plan,
replacement recommendation, and rollback controls.

The separately authorised cleanup operation is also staging-only. It requires
`ALLOW_PILOT_CANONICAL_CLEANUP=1`, the literal confirmation token
`CLEANUP-PILOT-CANONICAL-DATASET-V1`, and the exact reviewed manifest hash.
It uses the sanctioned regulated-deletion context and refuses any mismatched,
non-fixture, or unexpected `RM-PILOT-*` identity. Nothing invokes cleanup
automatically.

## How to run

From `arie-backend/`:

```bash
# List all 23 scenarios in the registry (including 12 Item 36 states)
python -m fixtures.cli list

# Seed everything inside a transaction and roll back (safe anywhere)
python -m fixtures.cli dry-run

# Seed only specific scenarios
python -m fixtures.cli dry-run --scen SCEN-01,SCEN-02

# Render the register (Markdown table) - dry-run, no commit
python -m fixtures.cli register --out fixtures/REGISTER.md

# SCEN-05 fixture validation (no writes, no env gate) - confirms the explicit
# SCEN-05 registry contract: completed periodic review with no memo and no
# EDD. Also runs at the end of dry-run / apply.
python -m fixtures.cli check

# Apply against a real DB (staging only - triple-gated)
ENVIRONMENT=staging \
ALLOW_FIXTURE_SEED=1 \
python -m fixtures.cli apply --confirm I-UNDERSTAND-FIXTURE-WRITE
```

## Safety gates (apply mode only)

All three of these must be true or `apply` refuses:

1. `ENVIRONMENT=staging`
2. `ALLOW_FIXTURE_SEED=1`
3. `--confirm I-UNDERSTAND-FIXTURE-WRITE`

`dry-run`, `list`, `register`, and `check` have no environment gate.

## Real-schema mapping

The original draft package referenced columns that do not exist in the
live schema. Path A folds those semantics into existing TEXT columns
using deterministic markers, so fixture intent is preserved without
schema churn.

| Drafted field | Real storage | Notes |
|---|---|---|
| `applications.risk_level` (lowercase) | UPPERCASE | Schema CHECK requires UPPERCASE |
| `applications.status='active'` | `'in_review'` | `'active'` is not in the CHECK enum |
| `applications.id` | SCEN-01..11 use `f1xed...`; Item 36 uses generated TEXT ids | Item 36 identity is its fixture key + marker + synthetic ref + `is_fixture`, not a global PK |
| `applications.ref` | `FX-ITEM36-*` for Item 36 | Reserved synthetic namespace; normal application generation remains `ARF-*` |
| `monitoring_alerts.dismissal_payload` | `officer_notes` with `FIX_PAYLOAD_JSON:` sentinel | Recoverable via `fixtures.seeder.parse_dismissal_payload` |
| `monitoring_alerts.updated_at` | (omitted) | Column doesn't exist on PG |
| `periodic_reviews.status` | `trigger_type` | Free text in both schemas |
| `periodic_reviews.review_memo` | packed into `trigger_reason` after `FIX_REVIEW_JSON:` sentinel as JSON | Recoverable via `fixtures.seeder.parse_review_payload`. Round-trip safe regardless of memo content (no `;`/`=` parsing) |
| `periodic_reviews.outcome` | `decision` (also mirrored in JSON payload) | |
| `periodic_reviews.source_alert_id` | packed into `trigger_reason` JSON payload | |
| `periodic_reviews.updated_at` | (omitted) | Column doesn't exist on PG |
| `edd_cases.kind` | `trigger_source` (also mirrored in JSON payload) | Free text |
| `edd_cases.status` | `stage` | Constrained enum; `'information_gathering'` is the safe default |
| `edd_cases.memo_id` | (omitted) | Memo lookup is by `application_id` |
| `edd_cases.source_review_id` | packed into `trigger_notes` after `FIX_EDD_JSON:` sentinel as JSON | Recoverable via `fixtures.seeder.parse_edd_payload` |
| `edd_cases.source_alert_id` | packed into `trigger_notes` JSON payload | |
| `compliance_memos.reference` | embedded in `memo_data` JSON | |
| `compliance_memos.title` / `body` | embedded in `memo_data` JSON | `memo_data` is the only required text column |
| `compliance_memos.status` | `review_status='approved'` | Existing canonical column |
| `compliance_memos.updated_at` | (omitted) | Column doesn't exist |
| `documents.purpose` | `doc_type` | |

## Item 36 negative-path manifests

`fixtures.registry.NEGATIVE_PATH_FIXTURES` is the machine-readable contract
for SCEN-12 through SCEN-23. Each entry declares its stable fixture key and
synthetic marker, expected state/control/HTTP result, every table written,
regulated tables, teardown order, and whether founder-approved staging
retention may be considered.

Item 36 refs are collision-preflighted for all selected roots before any root,
client, child, or evidence write. A matching identity is updated idempotently;
a non-fixture owner or mismatched fixture identity fails the whole transaction.
The cross-client pair is preflighted as two explicit roles under the same key.

The expected HTTP statuses come from the existing
gate suites; the fixture tests invoke the exact non-mutating backend gate/helper
where an API decision call could mutate evidence.

| Scenario | Fixture key | Expected control | HTTP | Retain option |
|---|---|---|---:|:---:|
| SCEN-12 | `active-approval-blockers` | approval gate blockers | 400 | yes |
| SCEN-13 | `stale-memo` | stale memo gate | 400 | no |
| SCEN-14 | `stale-risk-provenance` | stale risk provenance gate | 409 | no |
| SCEN-15 | `missing-idv` | unresolved IDV gate | 400 | yes |
| SCEN-16 | `missing-required-documents` | document reliance gate | 400 | yes |
| SCEN-17 | `pending-rmi` | RMI continuation gate | 409 | no |
| SCEN-18 | `synthetic-sanctions-hit` | prohibited screening hit | 400 | yes |
| SCEN-19 | `outstanding-pep-review` | declared-PEP compliance route | 400 | yes |
| SCEN-20 | `periodic-review-blockers` | periodic review readiness | 409 | yes |
| SCEN-21 | `completed-periodic-review` | completed review read | 200 | no |
| SCEN-22 | `similar-reference-cross-client` | cross-client ownership denial | 403 | no |
| SCEN-23 | `already-consumed-approval` | decision replay denial | 409 | no |

## Audit trail

Every mutating operation writes a `fixture.<action>` row to
`audit_log` with the canonical column set:

```
user_id        = 'fixture_seed'
user_name      = 'fixture_seed'
user_role      = 'system'
action         = 'fixture.<insert_application|update_alert|...>'
target         = '<table>:<id>'
detail         = human-readable line
ip_address     = '127.0.0.1'
before_state   = JSON or NULL
after_state    = JSON or NULL
```

This matches the column shape used by every existing `INSERT INTO
audit_log` site in `server.py` and `base_handler.py`. The
`before_state` / `after_state` columns were added by inline migration
v2.18 (`db.py:2458`).

## Known gaps vs the original draft package

These are **deliberate fidelity losses** accepted under Path A in
exchange for zero schema churn:

- **SCEN-06 dismissal payload** lives as JSON inside `officer_notes`,
  not in a dedicated `dismissal_payload` JSONB column. Recover with
  `fixtures.seeder.parse_dismissal_payload`.
- **Alert→Review→EDD linkage** is sentinel-JSON encoded in
  `periodic_reviews.trigger_reason` and `edd_cases.trigger_notes`
  (see mapping table). The leading `FIX_SCENxx_<KIND>` marker remains a
  plain prefix so `LIKE` lookups still work, but `source_alert_id` and
  `source_review_id` are inside the JSON payload — not first-class FK
  columns. Recoverable via `parse_review_payload` / `parse_edd_payload`.
- **periodic_reviews.review_memo** is JSON-encoded inside
  `trigger_reason`. The PG schema has no `review_memo` column.
  Renderers parse it back via `parse_review_payload`.
- **Compliance memo `reference` / `title` / `body`** live as JSON
  inside `memo_data`, the only required text column on the table.
- **SCEN-05** is explicitly seeded as the negative control / no-memo
  counterpart to SCEN-04: a completed periodic review with
  `review_memo=None` and no `edd_spec` (so no compliance memo is created
  by the seeder either). The `python -m fixtures.cli check` subcommand
  validates the SCEN-05 registry contract (completed, no memo, no EDD).
  The check also runs at the end of `dry-run` and `apply`.
- **SCEN-11 outstanding alert** is rendered in the alert pane only,
  not as a generated required-item. Reflects current product behaviour.

## Workflow

`.github/workflows/seed-staging-fixtures.yml` is included as a
scaffold. It still contains operational placeholders that need real
values (`REGMIND_STAGING_TASK_FAMILY`, the Fargate artifact-extraction

workflow is safe by default: it only invokes `dry-run` unless an
operator manually edits the dispatch input.

## Cleanup

Item 36 cleanup is intentionally available only through the existing P12-1
`fixture_cleanup_nonprod` sanctioned context. `cleanup_registered_fixture`
accepts one registered fixture key, refuses non-fixture application roots, and
deletes only exact marker/ID predicates in that manifest's declared order. It
does not provide a generic table wipe, and it does not alter P12-1 protections.

From `arie-backend/`, with the main DB configuration explicitly pointed at the
intended non-production database:

```bash
ENVIRONMENT=staging python - <<'PY'
from db import get_db
from fixtures.cleanup import cleanup_registered_fixture

db = get_db()
try:
    print(cleanup_registered_fixture(
        db,
        "active-approval-blockers",
        actor_id="operator-id",
        confirmation="DELETE-SYNTHETIC-FIXTURE",
    ))
finally:
    db.close()
PY
```

Direct `DELETE` against regulated fixture evidence remains denied. For the
post-merge staging walkthrough, seed via the existing CLI, demonstrate the
blocked approval/replay/cross-client controls, and then present the founder
with either (A) cleanup of all Item 36 fixtures or (B) retention only of entries
whose manifest says `safe_to_retain_in_staging=True`. Do not infer that choice.
